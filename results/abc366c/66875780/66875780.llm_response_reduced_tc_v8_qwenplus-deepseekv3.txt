To fix this code, we need to address the issue where the set is not correctly tracking the number of balls with each integer. The current implementation only keeps track of whether an integer is present in the bag, but not how many balls of each integer there are. This leads to incorrect behavior when multiple balls of the same integer are added and removed.

### Approaches
1. **Track Counts**: Use a dictionary (like `unordered_map` in C++) to keep track of the count of each integer in the bag. This allows us to accurately add and remove balls without losing track of duplicates.
2. **Track Unique Integers**: Use a set to keep track of the unique integers present in the bag. The size of this set will give the answer for query type 3.
3. **Update Accordingly**: For each query:
   - **Type 1 (Add x)**: Increment the count of `x` in the dictionary. If the count becomes 1, add `x` to the set.
   - **Type 2 (Remove x)**: Decrement the count of `x` in the dictionary. If the count becomes 0, remove `x` from the set.
   - **Type 3 (Print Unique Count)**: The size of the set is the answer.

### Explanation
- **Tracking Counts**: The `countMap` (unordered_map) keeps a count of how many balls of each integer are in the bag. This allows us to correctly handle multiple additions and removals of the same integer.
- **Tracking Unique Integers**: The `uniqueSet` (set) maintains only the integers that have at least one ball in the bag. This ensures that the size of the set accurately reflects the number of different integers present.
- **Query Handling**: 
  - **Type 1 (Add x)**: Increment the count for `x`. If the count reaches 1, add `x` to the set.
  - **Type 2 (Remove x)**: Decrement the count for `x`. If the count drops to 0, remove `x` from the set.
  - **Type 3 (Print Unique Count)**: Simply output the size of the set, which is the number of different integers in the bag.

This approach efficiently handles all operations in logarithmic time for the set operations, making it suitable for the given constraints.
