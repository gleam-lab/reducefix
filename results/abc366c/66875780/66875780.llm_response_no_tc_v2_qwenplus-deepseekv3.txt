The issue with the original code is that it uses a `set` to keep track of the distinct integers in the bag. However, this approach is incorrect because a `set` doesn't account for the multiplicity of each integer. For example, if you have multiple balls with the same integer, the `set` will only store one instance, and removing one ball with that integer will remove it entirely from the set, even if there are still other balls with the same integer left in the bag.

To fix this, we need to use a data structure that can keep track of both the integers and their counts. A `map` (or `unordered_map` for better performance) is suitable for this purpose. Here's the corrected code:

```cpp
#include <iostream>
#include <unordered_map>
#include <set>

using namespace std;

void fastIO() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    cout.tie(nullptr);
}

void Solve() {
    int Q;
    cin >> Q;
    unordered_map<int, int> count_map;
    int distinct_count = 0;
    
    while (Q--) {
        int type;
        cin >> type;
        
        if (type == 1) {
            int x;
            cin >> x;
            if (count_map[x] == 0) {
                distinct_count++;
            }
            count_map[x]++;
        } else if (type == 2) {
            int x;
            cin >> x;
            count_map[x]--;
            if (count_map[x] == 0) {
                distinct_count--;
            }
        } else {
            cout << distinct_count << "\n";
        }
    }
}

int main() {
    fastIO();
    Solve();
    return 0;
}
```

### Explanation:
1. **Data Structures**: We use an `unordered_map<int, int>` to keep track of the count of each integer in the bag. This allows O(1) average time complexity for insertions, deletions, and updates.
2. **Distinct Count**: We maintain a separate variable `distinct_count` to keep track of the number of distinct integers in the bag. This variable is incremented when a new integer is added (count changes from 0 to 1) and decremented when the count of an integer drops to 0.
3. **Efficiency**: The operations for each query are O(1) on average, making the solution efficient for the given constraints (Q up to 2×10^5).

This approach correctly handles the multiplicity of integers and ensures that the count of distinct integers is accurately reported for each type 3 query.
