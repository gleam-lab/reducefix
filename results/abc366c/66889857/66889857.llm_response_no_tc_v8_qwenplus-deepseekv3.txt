The initial code provided uses a `set<int>` to track the unique integers in the bag. While this works for counting the number of different integers (query type 3), it fails for query type 2 (removing a ball with integer x) because it removes all instances of x from the set, not just one. The bag can contain multiple balls with the same integer, so we need to track the count of each integer separately.

To fix this, we should use a `map<int, int>` or an array to keep track of the count of each integer in the bag. Here's the corrected code:

```cpp
#include <bits/stdc++.h>
using namespace std;
#define rep(i, n) for (int i = 0; i < (int)(n); i++)
#define rep2(i, s, n) for (int i = (s); i < (int)(n); i++)

int main() {
    int Q; cin >> Q;
    unordered_map<int, int> count;
    set<int> unique_numbers;

    rep(i, Q) {
        int query_type; cin >> query_type;
        if (query_type == 1) {
            int x; cin >> x;
            count[x]++;
            if (count[x] == 1) {
                unique_numbers.insert(x);
            }
        } else if (query_type == 2) {
            int x; cin >> x;
            count[x]--;
            if (count[x] == 0) {
                unique_numbers.erase(x);
            }
        } else if (query_type == 3) {
            cout << unique_numbers.size() << endl;
        }
    }
    return 0;
}
```

### Explanation:
1. **Data Structures**: 
   - `unordered_map<int, int> count` keeps track of how many balls of each integer `x` are in the bag.
   - `set<int> unique_numbers` maintains the unique integers in the bag, which is needed for the third query.

2. **Query Handling**:
   - **Type 1 (`1 x`)**: Increment the count of `x` in the map. If the count becomes 1, add `x` to the set (since it's now present in the bag).
   - **Type 2 (`2 x`)**: Decrement the count of `x` in the map. If the count drops to 0, remove `x` from the set (since it's no longer in the bag).
   - **Type 3 (`3`)**: Output the size of the set, which gives the number of different integers in the bag.

This approach ensures that each operation is handled efficiently, with O(1) average time complexity for insertions, deletions, and lookups (using an unordered_map and set), making it suitable for large input sizes (up to 2×10^5 queries).
