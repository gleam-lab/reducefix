After analyzing the problem and the failing test case, I notice that the issue isn't with the algorithm itself but rather with how the input is being read. The original code uses a single `cin >> x` to read the query type, which can cause problems when processing the input. For queries of type 2 and 3, we need to properly handle the input reading to avoid mixing up the values.

Here's the corrected C++ code that properly handles all query types:

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(nullptr);
    
    int Q;
    cin >> Q;
    unordered_map<int, int> count;
    int distinct = 0;
    
    while (Q--) {
        int type;
        cin >> type;
        if (type == 1) {
            int x;
            cin >> x;
            if (count[x] == 0) distinct++;
            count[x]++;
        } else if (type == 2) {
            int x;
            cin >> x;
            if (count[x] > 0) {
                count[x]--;
                if (count[x] == 0) distinct--;
            }
        } else if (type == 3) {
            cout << distinct << '\n';
        }
    }
    return 0;
}
```

Key improvements:
1. Uses `unordered_map` to track counts of each number, which allows O(1) operations on average
2. Maintains a `distinct` counter that's updated when numbers are added/removed
3. Properly handles the input reading for all query types
4. Uses faster I/O with `ios_base::sync_with_stdio(false)` and `cin.tie(nullptr)`
5. Ensures correct output format with `'\n'` instead of `endl`

The solution efficiently tracks the number of distinct integers in the bag by:
- Incrementing the distinct count when a new number is added (count goes from 0 to 1)
- Decrementing the distinct count when a number's count drops to 0
- For query type 3, simply outputs the current distinct count

This approach ensures all operations are O(1) on average, making it efficient enough for the given constraints (Q ≤ 2×10^5).
