The issue with the original code is that it uses a `set` to track the distinct integers, which works correctly for counting distinct values, but the problem requires handling multiple occurrences of the same integer (which the original code doesn't account for). The test case shows that the output should correctly reflect the number of distinct integers after each operation, even when multiple balls with the same integer are present.

To fix this, we need to track the count of each integer in the bag. We can use a `map` or `unordered_map` to keep track of the count of each integer. When adding a ball, we increment the count, and when removing, we decrement it. If the count reaches zero, we remove the integer from the map. The size of the map will always give the number of distinct integers.

Here's the corrected code:

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(nullptr);
    
    unordered_map<int, int> count_map;
    int distinct_count = 0;
    int q;
    cin >> q;
    
    while (q--) {
        int type;
        cin >> type;
        
        if (type == 1) {
            int x;
            cin >> x;
            if (count_map[x] == 0) {
                distinct_count++;
            }
            count_map[x]++;
        } else if (type == 2) {
            int x;
            cin >> x;
            if (count_map[x] > 0) {
                count_map[x]--;
                if (count_map[x] == 0) {
                    distinct_count--;
                }
            }
        } else if (type == 3) {
            cout << distinct_count << "\n";
        }
    }
    
    return 0;
}
```

### Explanation:
1. **Data Structure**: We use an `unordered_map` to keep track of the count of each integer in the bag. This allows O(1) average time complexity for insertions, deletions, and lookups.
2. **Distinct Count**: We maintain a separate variable `distinct_count` to keep track of the number of distinct integers. This avoids the need to compute the size of the map each time a query of type 3 is encountered.
3. **Operations**:
   - **Type 1 (Add x)**: Increment the count of `x`. If it was previously zero, increment `distinct_count`.
   - **Type 2 (Remove x)**: Decrement the count of `x`. If it reaches zero, decrement `distinct_count`.
   - **Type 3 (Print distinct count)**: Simply output the value of `distinct_count`.
4. **Efficiency**: The operations are O(1) on average, making the solution efficient for the given constraints (Q ≤ 2 × 10^5).

This approach ensures that we correctly handle multiple occurrences of the same integer while efficiently maintaining and querying the count of distinct integers.
