To solve this problem, we need to manage a dynamic undirected graph with two types of operations:

1. **Union Operation (`Type 1`)**: Connect two nodes.
2. **Query Operation (`Type 2`)**: Return the k-th largest vertex in the connected component of a given node.

The challenge is efficiently handling up to `2 * 10^5` queries while maintaining fast access to the k-th largest node in each connected component.

---

### Key Observations:
- We can use **Disjoint Set Union (DSU/Union-Find)** for managing connectivity.
- For efficient access to the k-th largest element, we maintain an ordered set using `ordered_set` from Policy-Based Data Structures (`<ext/pb_ds/...>`), which supports `k-th` largest/smallest in O(log N) time.

---

### Issues in the Original Code:
- The union logic was flawed by limiting the number of elements inserted into the smaller set (e.g., `if(V[v].size() > 30) break;`). This led to missing elements in some sets.
- This caused incorrect results when querying for larger `k`, like `k=7`, because the set didn't have all the required vertices.
- Additionally, inserting only a limited subset during union breaks the correctness and completeness of the data structure.

---

### Fix Summary:
- Remove any limit on how many elements are copied during the union.
- Always merge the smaller set into the larger one to ensure amortized efficiency (small-to-large merging).
- Ensure that after unioning, both components' vertices are fully represented in the new set.

---

### Final Optimized C++ Code:
```cpp
#include <iostream>
#include <vector>
#include <ext/pb_ds/assoc_container.hpp>
#include <ext/pb_ds/tree_policy.hpp>

using namespace std;
using namespace __gnu_pbds;

typedef tree<int, null_type, less<int>, rb_tree_tag, tree_order_statistics_node_update> ordered_set;

const int MAXN = 2 * 100000 + 10;

int parent[MAXN];
ordered_set components[MAXN];

int find(int x) {
    if (parent[x] != x)
        parent[x] = find(parent[x]);
    return parent[x];
}

void unite(int u, int v) {
    int pu = find(u);
    int pv = find(v);
    if (pu == pv)
        return;

    // Merge smaller set into larger set
    if (components[pu].size() < components[pv].size())
        swap(pu, pv);

    for (int x : components[pv]) {
        components[pu].insert(x);
    }

    parent[pv] = pu;
}

int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(nullptr);

    int N, Q;
    cin >> N >> Q;

    // Initialize DSU and ordered sets
    for (int i = 1; i <= N; ++i) {
        parent[i] = i;
        components[i].insert(i); // Store actual values
    }

    while (Q--) {
        int type;
        cin >> type;

        if (type == 1) {
            int u, v;
            cin >> u >> v;
            unite(u, v);
        } else {
            int v, k;
            cin >> v >> k;
            int root = find(v);
            if (components[root].size() < k) {
                cout << -1 << "\n";
            } else {
                // k-th largest = size - k-th smallest in descending order
                cout << *components[root].find_by_order(components[root].size() - k) << "\n";
            }
        }
    }

    return 0;
}
```

---

### Explanation:
- Each node starts as its own parent.
- Each node's value is stored in its own `ordered_set`.
- When uniting two components:
  - We merge the smaller set into the larger one to keep complexity low.
  - All elements from the smaller set are added to the larger one.
- For queries:
  - Find the root of the component.
  - Use `find_by_order` to get the k-th largest element (by computing index from the end).

---

### Complexity:
- Time per operation:
  - Union: Amortized O(log N) due to small-to-large merging.
  - Query: O(log N) due to `ordered_set`.

This approach ensures correctness and handles large inputs efficiently.
