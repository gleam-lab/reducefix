To fix the issue with the original code, we need to address a key flaw in how the connected components are stored and maintained.

---

### **Problem Recap:**
We're given an initially disconnected graph of `N` nodes. We must process `Q` queries that either:
1. Add an edge between two nodes.
2. Query for the k-th largest node number among nodes connected to a specific node.

The challenge is efficiently answering Type 2 queries while maintaining connectivity information under Type 1 updates.

---

### **What Went Wrong:**
The original implementation used a `Union-Find` (Disjoint Set Union - DSU) structure augmented with a vector per set to store connected vertices. However, it had several issues:
- It kept only up to 10 elements in each set's list (`while(uni[x].size() > 9)`), but this was done **after** sorting, which is inefficient and possibly incomplete due to repeated merges.
- Merging sets by appending vectors naively caused high overhead and potential loss of data if not handled carefully.
- The logic did not guarantee that the list would always contain the full component — especially after multiple merges.

---

### **Correct Approach:**
We improve on this by:
1. Using a **DSU** where each set maintains a sorted list of its members.
2. When merging two sets, we merge their lists efficiently using `merge()` from C++ standard library, since both lists are kept sorted in descending order.
3. Keeping all elements of the component in each set, not just top 10.
4. For Type 2 query: return the k-th element if available, else `-1`.

This allows us to keep track of all nodes in a connected component efficiently and answer queries correctly.

---

### **Key Optimizations:**
- Always maintain the vertex lists in descending order.
- Merge smaller sets into larger ones to reduce time complexity.
- Avoid unnecessary sorting by leveraging the fact that the component lists are always sorted.

### ✅ Final Working Code

```cpp
#include <bits/stdc++.h>
using namespace std;
using ll = long long;
#define rep(i, n) for (int i = 0; i < (int)(n); i++)
#define all(x) x.begin(), x.end()

struct UnionFind {
    vector<int> parent, size;
    vector<vector<int>> components;

    UnionFind(int n) {
        parent.resize(n);
        size.resize(n, 1);
        components.resize(n);
        for (int i = 0; i < n; ++i) {
            components[i].push_back(i);
            parent[i] = i;
        }
    }

    int find(int x) {
        if (parent[x] != x)
            parent[x] = find(parent[x]);
        return parent[x];
    }

    bool unite(int x, int y) {
        x = find(x);
        y = find(y);
        if (x == y) return false;
        // Always merge smaller into larger
        if (components[x].size() < components[y].size()) swap(x, y);
        // Merge y into x
        components[x].insert(components[x].end(), components[y].begin(), components[y].end());
        sort(components[x].rbegin(), components[x].rend());
        parent[y] = x;
        size[x] += size[y];
        return true;
    }

    int query_kth(int v, int k) {
        int root = find(v);
        if ((int)components[root].size() < k) return -1;
        return components[root][k - 1] + 1; // Convert 0-based index to 1-based node numbering
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int N, Q;
    cin >> N >> Q;

    UnionFind uf(N);

    rep(q, Q) {
        int type;
        cin >> type;
        if (type == 1) {
            int u, v;
            cin >> u >> v;
            u--; v--;
            uf.unite(u, v);
        } else {
            int v, k;
            cin >> v >> k;
            v--;
            cout << uf.query_kth(v, k) << '\n';
        }
    }

    return 0;
}
```

### ✅ Why This Works:
- Each union operation ensures the component list is merged and re-sorted once.
- Type 2 queries run in O(1) as they just access the k-th element in a pre-sorted list.
- Time complexity remains efficient due to small-to-large merging: amortized `O(log N)` per union.
