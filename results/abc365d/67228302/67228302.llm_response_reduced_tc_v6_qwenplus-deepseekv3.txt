After analyzing the problem and the failing test case, it seems the original code was trying to count the minimum number of moves needed to beat a sequence of Rock-Paper-Scissors moves, but it had a logical flaw in how it determined when to increment the counter.

Here's the corrected solution that properly counts the number of optimal moves needed to beat the sequence:

```cpp
#include <bits/stdc++.h>
using namespace std;
#define endl "\n"
#define MOD 1000000007

int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(0);

    map<char, char> beats = {{'R', 'P'}, {'P', 'S'}, {'S', 'R'}};
    int n;
    cin >> n;
    string s;
    cin >> s;
    
    if (n == 0) {
        cout << 0 << endl;
        return 0;
    }
    
    int total = 1;
    char current_move = beats[s[0]];
    
    for (int i = 1; i < n; i++) {
        if (s[i] == s[i-1]) {
            continue;
        }
        if (beats[s[i]] != current_move) {
            current_move = beats[s[i]];
            total++;
        }
    }
    
    cout << total << endl;
    
    return 0;
}
```

Key changes made:
1. The logic now properly checks if consecutive moves are the same (in which case we don't need to change our move)
2. Only increments the counter when we need to change our move to beat the current opponent's move
3. Added handling for empty input case
4. The algorithm now correctly counts transitions between different moves needed to beat the sequence

For the test case "PPR":
- First 'P' is beaten by 'S' (count = 1)
- Second 'P' is the same as first, so no change needed
- 'R' requires 'P', which is different from current 'S', so we switch and increment count (count = 2)
This matches the expected output of 2.
