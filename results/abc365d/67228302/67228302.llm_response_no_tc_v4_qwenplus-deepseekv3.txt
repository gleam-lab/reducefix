After analyzing the code, I found that the logic for counting the number of moves needed to beat the sequence of Rock-Paper-Scissors (RPS) moves is incorrect. The current implementation doesn't properly account for consecutive moves that can be beaten by the same counter move. Here's the fixed version:

```cpp
#include <bits/stdc++.h>
using namespace std;
#define endl "\n"
#define MOD 1000000007

int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(0);

    map<char, char> beats = {{'R', 'P'}, {'P', 'S'}, {'S', 'R'}};
    int n;
    cin >> n;
    string s;
    cin >> s;
    
    if (n == 0) {
        cout << 0 << endl;
        return 0;
    }
    
    int total = 1;
    char current_beat = beats[s[0]];
    
    for (int i = 1; i < n; i++) {
        if (beats[s[i]] != current_beat) {
            current_beat = beats[s[i]];
            total++;
        }
    }
    
    cout << total << endl;
    
    return 0;
}
```

Key fixes:
1. Changed the logic to only increment the counter when we need to switch to a different move to beat the current RPS move
2. Simplified the condition check - we only need to change our move if the optimal move for the current input differs from our current move
3. Added a check for empty input (though the problem likely guarantees n ≥ 1)
4. Renamed `last_move` to `current_beat` for better clarity
5. The fixed version properly counts the minimum number of moves needed to beat the entire sequence

The algorithm now correctly counts the number of distinct moves needed to beat all consecutive groups of the same move in the input sequence. For example:
- "RRRP" would require 2 moves (P for RRR, then S for P)
- "RRRSSSPPP" would require 3 moves (P for R, S for S, R for P)
