To fix this code, we need to correctly calculate the maximum number of changes in strategy that can lead to winning moves in a Rock-Paper-Scissors game. The previous approach had some logical flaws in tracking the last move and counting the changes. 

### Approaches
1. **Initial Analysis**: The problem involves determining the optimal sequence of moves to maximize the number of times a player can beat the opponent's moves in a sequence. The player can choose to either beat the current move or the next move, which requires careful consideration of transitions between moves.
2. **Key Insight**: The optimal strategy involves making a move that beats the current or the next opponent's move, whichever allows for more transitions (changes) that result in a winning move.
3. **Algorithm Selection**: We'll iterate through the sequence, checking each move and deciding whether to beat the current move or the next move, based on which choice allows for more transitions. This ensures we maximize the number of winning moves.

### Solution Code
```cpp
#include <bits/stdc++.h>
using namespace std;
#define endl "\n"
#define MOD 1000000007

int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(0);

    map<char, char> beats = {{'R', 'P'}, {'P', 'S'}, {'S', 'R'}};
    int n;
    cin >> n;
    string s;
    cin >> s;
    if (n == 0) {
        cout << 0 << endl;
        return 0;
    }
    
    int total = 1;
    char last_move = beats[s[0]];
    for (int i = 1; i < n; ) {
        if (i + 1 < n && beats[s[i]] != beats[s[i+1]]) {
            if (beats[s[i+1]] == last_move) {
                last_move = beats[s[i]];
                total++;
                i++;
            } else {
                last_move = beats[s[i+1]];
                total++;
                i += 2;
            }
        } else {
            if (beats[s[i]] != last_move) {
                last_move = beats[s[i]];
                total++;
            }
            i++;
        }
    }
    cout << total << endl;

    return 0;
}
```

### Explanation
1. **Initialization**: We read the input values and initialize the map `beats` which defines which move beats another (e.g., 'R' is beaten by 'P').
2. **Edge Case Handling**: If the input string is empty, we immediately return 0 as there are no moves to consider.
3. **Iterating Through Moves**: We start with the first move and decide the initial strategy to beat it. For subsequent moves, we check if beating the current move or the next move would allow more transitions that result in a winning move.
4. **Transition Logic**: If the next move is different from the current move, we decide to beat the move that allows for a transition without conflict. If they are the same, we simply beat the current move if it hasn't been beaten already by the last move.
5. **Output the Result**: The total count of winning transitions is printed, representing the maximum number of times the player can beat the opponent's moves in the sequence.

This approach ensures that we maximize the number of winning moves by strategically choosing to beat either the current or the next move, based on the optimal transition path.
