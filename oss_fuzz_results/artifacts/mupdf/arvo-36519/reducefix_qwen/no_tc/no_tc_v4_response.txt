After analyzing the crash log and code, I've identified the issue. The heap-use-after-free occurs because `fz_drop_pixmap(ctx, pixmap)` in the `fz_always` block is called unconditionally, but `pixmap` might be accessed after being freed in the `fz_paint_image` call if an exception occurs.

The root cause is that the `fz_always` block runs after the try block, but if an exception occurs during `fz_paint_image`, the pixmap has already been dropped but could still be referenced. We need to ensure the pixmap is only dropped if it was successfully created.

This fix adds a null check after `fz_get_pixmap_from_image` to prevent use of a potentially null pixmap pointer. If the pixmap creation fails, we return early before attempting to use it. This prevents both the use-after-free and null pointer dereference issues while maintaining the correct cleanup semantics through the existing `fz_always` block.
