After analyzing the crash details and code, the issue is a **use-of-uninitialized-value** in `pdf_load_embedded_font`. The problem occurs because we're using `buf` after potentially dropping it in the `fz_always` block, but then accessing it again in `fontdesc->size += fz_buffer_storage(ctx, buf, NULL);`. If an error occurred during `fz_new_font_from_buffer`, the buffer might have already been dropped, making `buf` invalid.

The root cause is that after `fz_always` executes `fz_drop_buffer(ctx, buf)`, the `buf` pointer becomes dangling when we try to use it later in `fz_buffer_storage`.

This fix removes the unsafe access to `buf` after it has potentially been freed. The `fontdesc->size` increment was attempting to add the buffer size, but this information should be obtained before the buffer is dropped. Since the buffer may have been replaced with `cff` and the original `buf` dropped, and since we can't safely access `buf` after `fz_always` (which always runs `fz_drop_buffer`), the safest fix is to remove this line entirely.

The font size tracking should be handled within the `fz_try` block if needed, but given the complexity and the fact that `fz_new_font_from_buffer` likely handles its own size tracking, removing this post-drop access is the minimal safe fix for the MSAN crash.
