The crash is a **Use-of-uninitialized-value** detected by MSAN (MemorySanitizer), occurring in `pdf_load_embedded_font`. The stack trace shows:

```
fz_drop_buffer
pdf_load_embedded_font
pdf_load_font_descriptor
```

This indicates that `fz_drop_buffer(ctx, buf)` is being called on a potentially uninitialized or already freed buffer. However, looking at the code:

- `buf` is assigned via `pdf_load_stream(ctx, stmref)`
- Then used in `fz_buffer_storage`, which may trigger use-of-uninitialized if `buf` is null or its contents are not fully initialized
- Later, even if `buf` is replaced with `cff`, the original pointer is dropped and `buf` reassigned — this is safe only if `buf` was valid
- Crucially: **if `pdf_load_stream` returns NULL**, then `fz_buffer_storage(ctx, buf, &data)` will dereference a null pointer indirectly through `buf`, but more importantly, MSAN may flag use of uninitialized memory inside the buffer

But wait: the real issue is likely **not null-pointer deref**, but rather **use of uninitialized memory within the buffer data** when calling `memcmp("OTTO", data, 4)`.

However, `fz_buffer_storage()` returns the size and sets `data = buf->data`. If the stream was loaded successfully but contains uninitialized bytes (e.g., due to incomplete I/O or padding), MSAN would complain here.

But the key insight is: **we're accessing `buf` after it has been dropped in the `fz_always` block, specifically in the line:**

```c
fontdesc->size += fz_buffer_storage(ctx, buf, NULL);
```

At this point, `buf` may have already been dropped via `fz_drop_buffer(ctx, buf)` in the `fz_always` block — which runs unconditionally *before* the catch/finally semantics take effect.

In MuPDF's exception model:
- `fz_try` / `fz_always` / `fz_catch` mimics try/finally/catch
- Code in `fz_always` runs **after** the `fz_try` block, **even if an exception occurred**
- So when we do `fz_drop_buffer(ctx, buf);` in `fz_always`, the buffer is freed
- Then, **after** `fz_always`, we execute:
  ```c
  fontdesc->size += fz_buffer_storage(ctx, buf, NULL);
  ```
  This accesses `buf` again — but it’s been dropped! Even though the pointer isn’t NULL, it points to freed memory → use-after-free equivalent, or at least use-of-dropped resource.

Moreover, `fz_buffer_storage(ctx, buf, NULL)` still needs to access the `fz_buffer` structure to read its `len`, so even just reading the size from a dropped buffer is invalid.

Thus, **the bug is**:  
We are using `buf` *after* it has been dropped in the `fz_always` block.

### ✅ Fix:
Move the `fontdesc->size += ...` line **before** the `fz_always`, so we read the buffer size before dropping it.

We must preserve all variables and macro pairings.
