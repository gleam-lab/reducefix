The crash is a **Use-of-uninitialized-value** detected by MSan (Memory Sanitizer), occurring in `pdf_load_embedded_font`. The stack trace shows the crash happens during buffer handling, specifically around `fz_drop_buffer`.

### 🔍 Root Cause Analysis

Looking at the code:

```c
buf = pdf_load_stream(ctx, stmref);

fz_try(ctx)
{
    size = fz_buffer_storage(ctx, buf, &data);
```

The issue arises **if `pdf_load_stream` fails and returns NULL**. In that case, `buf` is `NULL`, and calling `fz_buffer_storage(ctx, buf, &data)` leads to accessing an uninitialized or invalid buffer — but more critically, even if `buf` is valid, the structure of the error handling causes a **double use after free** or **use-after-drop** pattern.

However, the real problem is later:

```c
fz_always(ctx)
	fz_drop_buffer(ctx, buf);
```

This unconditionally drops `buf`, even if it was replaced inside the `fz_try` block (e.g., via `buf = cff`). But then, **after** the `fz_catch`, this line appears:

```c
fontdesc->size += fz_buffer_storage(ctx, buf, NULL);
```

Here’s the bug:  
- If an exception occurred during `fz_new_font_from_buffer`, the `fz_always` block runs and calls `fz_drop_buffer(ctx, buf)`, which sets `buf` to freed state.
- However, **after `fz_catch`**, we still try to access `buf` with `fz_buffer_storage(ctx, buf, NULL)` — **using a potentially already-freed or dangling buffer pointer**.

Moreover, because `fz_drop_buffer` may have already dropped the buffer, accessing it afterward is undefined behavior — and MSan flags this as **use-of-uninitialized-memory** because the memory might be marked as uninitialized after being freed.

But wait: there's another subtle issue.

If `pdf_extract_cff_subtable` succeeds, we do:
```c
fz_drop_buffer(ctx, buf);
buf = cff;
```
So far so good — but if `fz_new_font_from_buffer` throws an exception, then:
- `fz_always` runs → `fz_drop_buffer(ctx, buf)` → drops the `cff` buffer
- Then we fall through to `fz_catch` → `fz_rethrow`
- BUT THEN — after rethrowing, execution continues into:
```c
fontdesc->size += fz_buffer_storage(ctx, buf, NULL);
```

Which uses `buf` again — but `buf` now points to freed memory!

That means:
✅ **Bug**: After `fz_rethrow`, code *continues to execute*, but should not.  
❌ This function assumes control flow stops after `fz_rethrow`, but it doesn't — **C-style exceptions via `setjmp`/`longjmp` don’t stop execution unless you explicitly return or exit the scope.**

👉 Therefore, **any code after `fz_catch` will run even after an exception is caught and rethrown**, leading to use of dangling pointers.

Is that true?

No! Because `fz_catch(ctx) fz_rethrow(ctx);` means:
- If exception occurred → catch it → rethrow it → continue? ❌ Yes — in MuPDF’s model, `fz_rethrow` does NOT prevent fall-through.

In fact, **in mupdf**, after `fz_rethrow`, the next line is still executed unless you return.

See: https://github.com/ArtifexSoftware/mupdf/blob/master/source/fitz/exception.c

MuPDF uses `setjmp/longjmp`, so when you `fz_rethrow`, it eventually calls `longjmp`, meaning **any code after the `fz_catch` macro is skipped**.

But is that guaranteed?

Yes — `fz_rethrow` does a `longjmp` to the nearest `fz_try`, so **control never reaches code after `fz_catch` if an exception occurred**.

Therefore, the line:
```c

```
only runs **if no exception occurred during `fz_new_font_from_buffer`**.

So why is it crashing?

Because even if no exception occurred, `fz_always(ctx) fz_drop_buffer(ctx, buf);` **has already run**, which frees the buffer.

Then we call `fz_buffer_storage(ctx, buf, NULL)` — on a freed buffer!

Ah-ha! That’s the core issue.

Even in the **success case**, the `fz_always` block runs first — dropping `buf` — then we try to use it in the next line.

So the **execution order** is:

1. `fz_try`: create font → OK
2. `fz_always`: drop `buf` → buffer freed
3. (no exception) → skip `fz_catch`
4. Then run: `fontdesc->size += fz_buffer_storage(ctx, buf, NULL);` → use of freed buffer!

This is a **use-after-free**, which MSan detects as "use-of-uninitialized-value" because freed memory is considered uninitialized.
