After analyzing the crash details and code, the root cause is a **use-of-uninitialized-value** in `pdf_load_embedded_font`. The issue occurs because `buf` is used after being potentially dropped in the `fz_always` block when an exception occurs during `fz_new_font_from_buffer`.

The problem is in the line:
```c
fontdesc->size += fz_buffer_storage(ctx, buf, NULL);
```
Here, `buf` may have already been dropped by `fz_drop_buffer(ctx, buf)` in the `fz_always` block if an error occurred, making it invalid to use.

However, we cannot simply remove this line because `buf` declaration is needed later (as per instructions). The correct fix is to ensure we don't access `buf` after it's been dropped. Since the `fz_always` block always runs, and `fz_drop_buffer` invalidates `buf`, we need to check if `buf` is still valid before using it.

But looking more carefully: `fz_drop_buffer` doesn't set `buf` to NULL - it just frees the underlying data. The pointer itself becomes dangling. MSAN detects use of uninitialized memory because the buffer contents are used without guarantee they're initialized.

The real issue is that after `fz_always`, we're calling `fz_buffer_storage` on `buf` which may point to freed memory. We must re-load or preserve the size before the cleanup.

Actually, the correct fix is to capture the buffer size *before* the `fz_always` block, since after dropping we can't access it safely.

- `fz_buffer_storage(ctx, buf, NULL)` is called after `fz_drop_buffer(ctx, buf)` in `fz_always`
- Even though `fz_drop_buffer` is supposed to be safe to call multiple times, accessing buffer storage after drop leads to use-of-uninitialized-value
- We must move the size calculation before the cleanup
