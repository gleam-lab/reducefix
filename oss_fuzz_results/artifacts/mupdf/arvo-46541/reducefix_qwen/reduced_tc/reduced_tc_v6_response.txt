The crash is a **Use-of-uninitialized-value** detected by MSan (MemorySanitizer), occurring in `pdf_load_embedded_font`. The stack trace shows:

```
fz_drop_buffer
pdf_load_embedded_font
pdf_load_font_descriptor
```

This indicates that `fz_drop_buffer(ctx, buf)` is being called on an uninitialized or already-dropped buffer. Let's analyze the code flow.

---

### 🔍 Root Cause Analysis

Key lines:
```c
buf = pdf_load_stream(ctx, stmref);

fz_try(ctx)
{
    size = fz_buffer_storage(ctx, buf, &data); // <-- Uses 'buf'
    ...
    if (!memcmp("OTTO", data, 4)) {
        fz_buffer *cff = pdf_extract_cff_subtable(ctx, data, size);
        if (cff) {
            fz_drop_buffer(ctx, buf); // <-- Drops old buf
            buf = cff;                // <-- Reassigns buf
        }
    }

    fontdesc->font = fz_new_font_from_buffer(ctx, fontname, buf, 0, 1);
}
fz_always(ctx)
    fz_drop_buffer(ctx, buf); // <-- DROPS buf AGAIN?
```

**Problem**:  
If `pdf_load_stream` fails, it may return `NULL`. We do not check for this. Then:

- `fz_buffer_storage(ctx, buf, &data)` uses `buf` → use-of-uninitialized-or-null.
- Even worse: if `pdf_load_stream` returns `NULL`, then `fz_always` block calls `fz_drop_buffer(ctx, NULL)` — which might be safe in practice (if `fz_drop_buffer` tolerates NULL), but MSan reports use of uninitialized value *before* that point.

But more critically: **MSan says "use-of-uninitialized-value"**, not null pointer dereference. That suggests `buf` itself may be uninitialized *in some execution path*, or its contents are partially uninitialized.

Wait — actually, the variable `buf` is declared but not initialized:

```c
fz_buffer *buf;
...
buf = pdf_load_stream(ctx, stmref);
```

So if `pdf_load_stream` throws an exception **before assigning to `buf`**, then inside `fz_always`, we call `fz_drop_buffer(ctx, buf)` with an uninitialized pointer → **use-of-uninitialized-value**.

But `buf` is assigned immediately after declaration. So how could it be uninitialized during `fz_always`?

Ah! The key is understanding **how MuPDF’s `fz_try/fz_always/fz_catch` works**.

These are implemented using `setjmp`/`longjmp`. That means control can jump from deep inside `pdf_load_stream()` directly to the `fz_always` block — **without updating local variables that were modified before the jump**.

In particular, if `pdf_load_stream(ctx, stmref)` starts executing, modifies `buf` temporarily (e.g., in a nested try block), and then throws an exception, the final value of `buf` at the time of the longjmp may be garbage — because the assignment was interrupted mid-flight.

Thus: **the `buf` variable must be initialized to NULL to ensure that even if the assignment doesn't complete cleanly, the `fz_always` block sees a defined value.**

> This is a common pattern in C exception-handling systems: always initialize resource-holding locals to NULL when used across `fz_try`.

---

### ✅ Fix

Initialize `buf` to `NULL`.

We already have:
```c
fz_buffer *buf;
```

Change it to:
```c
fz_buffer *buf = NULL;
```

Also, since `fz_buffer_storage` is called unconditionally right after without checking `buf`, and `pdf_load_stream` may throw (which would go to `fz_catch` → `fz_rethrow`), we need to make sure we don’t access `buf` unless the operation succeeded.

But note: `pdf_load_stream` either returns a valid buffer or throws an exception. So if we reach inside `fz_try`, `buf` should be valid — unless there was a partial write due to longjmp.

However, the real issue reported by MSan is **uninitialized use**, not logic error. So initializing `buf` to `NULL` ensures that even if `longjmp` occurs during the assignment expression, `buf` has a defined value.

Even better: if an error occurs in `pdf_load_stream`, and control jumps into `fz_always`, we want to drop only if `buf` is non-null.

Actually, `fz_drop_buffer(ctx, NULL)` is safe — documented to do nothing. But **accessing an uninitialized pointer**, even to pass to a function, triggers MSan.

Therefore: **initialize `buf` to `NULL`**.
