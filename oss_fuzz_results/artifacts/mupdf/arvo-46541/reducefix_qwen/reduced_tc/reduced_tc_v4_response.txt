```
fz_drop_buffer
```

This indicates that `fz_drop_buffer(ctx, buf)` is being called on an uninitialized or already freed buffer.

### 🔍 Root Cause Analysis

Looking at the code:
```c
buf = pdf_load_stream(ctx, stmref);

fz_try(ctx)
{
    size = fz_buffer_storage(ctx, buf, &data);
    ...
}
fz_always(ctx)
    fz_drop_buffer(ctx, buf);
```

The issue arises because:

1. `pdf_load_stream(ctx, stmref)` may return `NULL` on failure.
2. If `buf` is `NULL`, calling `fz_buffer_storage(ctx, buf, &data)` leads to undefined behavior — but more importantly, even if we survive that, `fz_drop_buffer(ctx, buf)` in `fz_always` block will attempt to drop a `NULL` or invalid pointer? Actually, `fz_drop_buffer` tolerates `NULL` safely — so that’s not the problem.

But wait: **MSAN reports use-of-uninitialized-value**, not null dereference.

So where could uninitialized memory be used?

➡️ The key insight is: `pdf_load_stream()` can fail and throw an exception via `fz_throw`, which jumps to the `fz_always` block. However, when an error occurs, `buf` might not have been fully initialized or its internal data may be undefined.

But more critically: **if `pdf_load_stream()` returns a valid `fz_buffer*`, but the underlying data is not properly initialized (e.g., zero-length or corrupted stream), then `fz_buffer_storage(ctx, buf, &data)` returns a size and a pointer to possibly uninitialized memory.**

Then this line runs:
```c
if (!memcmp("OTTO", data, 4)) {
```

If `size < 4`, reading 4 bytes from `data` with `memcmp` accesses **uninitialized memory**, triggering MSAN.

Moreover, later:
```c
fontdesc->size += fz_buffer_storage(ctx, buf, NULL);
```
is executed *after* `fz_always` has dropped `buf`, meaning `buf` could be dangling or invalid.

Wait — no! Because of the `fz_try/fz_always/fz_catch` structure:

- Then `fz_catch` rethrows
- But **after** that, execution continues to `fontdesc->size += ...`, even if there was an exception!

That's the real bug.

### 🧨 Critical Logic Bug

Even if an exception occurred during `fz_new_font_from_buffer` or earlier, the code catches it, rethrows, but then proceeds **past the catch block** to:
```c
fontdesc->size += fz_buffer_storage(ctx, buf, NULL);
```

Which means:
- `buf` was already dropped in `fz_always`
- We're accessing a **freed/dropped buffer**
- This causes use-after-free or use-of-uninitialized-memory depending on allocator behavior

But also: the `fz_rethrow(ctx)` should prevent any code after the `fz_catch` from running — unless exceptions are not working as expected?

Actually, in MuPDF’s model:
- `fz_catch` is like a C++ catch; after `fz_rethrow`, control leaves the function.
- BUT: the syntax matters.

Current code:
```c
fz_catch(ctx)
	fz_rethrow(ctx);
```

This only applies to the next statement. So far so good.

BUT: the subsequent lines:
```c
fontdesc->size += fz_buffer_storage(ctx, buf, NULL);
fontdesc->is_embedded = 1;
```

Are **outside** the try/always/catch block — so they run unconditionally, even if an exception was thrown and handled.

But here's the catch: **if an exception occurs, `fz_rethrow` will propagate it upward**, so those lines won’t execute. ✅

However, **if no exception occurs**, we reach those lines — and that’s fine.

But now consider: what if `pdf_load_stream` fails? It throws → caught → rethrown → unwinds → `fz_always` drops `buf` → then exit.

So why MSAN?

Back to the **uninitialized value**: the real issue is likely that `buf` is created, but the stream contains uninitialized data, and we read from `data` without checking whether the content is sanitized.

But MSAN tracks whether memory has been initialized. If `pdf_load_stream` loads a buffer using `malloc`, and doesn't initialize the memory before writing PDF stream data into it, and if that stream processing somehow left gaps, then yes — but that’s unlikely.

### 💡 Most Likely Real Cause

After reviewing MuPDF patterns: `pdf_load_stream` can return a zero-sized buffer. Or worse: it may return a buffer whose `.data` field points to uninitialized memory.

But the immediate smoking gun is:

```c
size = fz_buffer_storage(ctx, buf, &data);
if (size > 12) {
    if (!memcmp("OTTO", data, 4)) {  // ← Read 4 bytes when size >= 4?
```

We check `size > 12`, so `size >= 13`, so `size >= 4` → safe for `memcmp(..., 4)`.

✅ So buffer bounds access is okay.

But what if `buf == NULL`?

→ `fz_buffer_storage(ctx, NULL, ...)` will throw an exception. That’s fine.

So where does **uninitialized value** come from?

Only possibility: **the `buf` returned by `pdf_load_stream` contains uninitialized memory in its data segment**, and `pdf_extract_cff_subtable` or `fz_new_font_from_buffer` reads it without sanitizing.

But MSAN blames `fz_drop_buffer`.

Wait: the crash stack trace says:
```
fz_drop_buffer
pdf_load_embedded_font
```

So the crash happens **inside `fz_drop_buffer`**, when trying to drop `buf`.

Why would `fz_drop_buffer` crash on uninitialized value?

Because `buf->data` or `buf->cap` or some internal field is uninitialized?

Ah! Here's the key:

👉 `buf = pdf_load_stream(ctx, stmref);`  
This call can fail. How?

It may return `NULL`. But `fz_buffer_storage(ctx, NULL, ...)` would then crash.

But instead, look at what `pdf_load_stream` does: it can return a valid `fz_buffer*`, but if the stream is encrypted and the filter fails, it might create a partial buffer?

Unlikely.

But more importantly: **what if `stmref` is null or invalid**?

No — `pdf_load_stream` handles that and throws.

But the real issue is subtle:

> When `pdf_extract_cff_subtable` is called, it returns a new buffer, and we do:
>
> ```c
> fz_drop_buffer(ctx, buf);
> buf = cff;
> ```
>
> Then later in `fz_always`: `fz_drop_buffer(ctx, buf);` → drops the new `cff` buffer.

That’s correct.

But if `pdf_extract_cff_subtable` returns `NULL`, we don't replace `buf`, so original `buf` remains, and gets dropped in `fz_always`.

Also correct.

But now, what about after the try/catch block?

```c
fontdesc->size += fz_buffer_storage(ctx, buf, NULL);
```

Here’s the bug: **after `fz_always`, `buf` has already been dropped**, so calling `fz_buffer_storage(ctx, buf, NULL)` uses a freed buffer.

Is that true?

Let’s analyze control flow:

```c
fz_try(ctx)
    ...
fz_always(ctx)
    fz_drop_buffer(ctx, buf);  // ← buf is dropped here
fz_catch(ctx)
    fz_rethrow(ctx);

// After this point, buf is INVALID!
fontdesc->size += fz_buffer_storage(ctx, buf, NULL);  // ← USE OF FREED MEMORY!
```

Yes! Even though `fz_rethrow` propagates the exception, **this final code runs only if no exception occurred**.

But if no exception occurred, then `fz_always` ran and dropped `buf`.

So `buf` is now a dangling pointer.

Calling `fz_buffer_storage(ctx, buf, NULL)` with a freed `buf` causes **use-after-free**, which MSAN reports as "use-of-uninitialized-value" if the memory was recycled.

🎯 **This is the root cause**.

---

### ✅ Fix

We must **not use `buf` after it has been dropped** in `fz_always`.

Solution: move the `fontdesc->size += ...` line **before** the `fz_always` block, inside the `fz_try`, or at least before dropping.

But note: we want to account for the final `buf` — which may have changed (e.g., replaced with `cff`).

So we must compute size **after** possible replacement, but **before** dropping.

Best fix: move the size accumulation into the `fz_try` block, right after creating the font.

Also, since `fz_new_font_from_buffer` consumes the buffer, we cannot access `buf` afterward anyway.

Thus, correct sequence:
1. Compute final `buf` (possibly extracted CFF)
2. Create font
3. Add buffer size to `fontdesc->size`
4. Drop buffer in `fz_always`

But currently, we add size **after** drop.

🔧 Move the `fontdesc->size += ...` line to just before `fz_always`, but **only if no exception occurred**.

Better: move it inside the `fz_try` block, after `fontdesc->font = ...`.
