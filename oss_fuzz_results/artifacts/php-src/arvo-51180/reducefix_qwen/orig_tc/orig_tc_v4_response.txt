### Bug Analysis

The crash is a **heap-use-after-free** in `update_errors_warnings`, triggered during `timelib_error_container_dtor` call. The stack trace shows:

```
timelib_error_container_dtor
php_date_initialize
zim_DateTime___construct
```

This indicates that after freeing an error container, it's being accessed again.

Looking at the code:
```c
if (DATEG(last_errors)) {
    timelib_error_container_dtor(DATEG(last_errors));  // Free existing
    DATEG(last_errors) = NULL;
}

// ... later ...

timelib_error_container_dtor(*last_errors);  // ⚠️ Use after free!
*last_errors = NULL;
```

### Root Cause

- If `DATEG(last_errors)` and `*last_errors` point to the **same** memory location, then the first `timelib_error_container_dtor(DATEG(...))` frees the memory.
- Then, later, `timelib_error_container_dtor(*last_errors)` tries to free the **same** memory again → **double free / use-after-free**.

Even worse: if `(*last_errors)` has warnings/errors, we assign it to `DATEG(last_errors)` *after* potentially destroying it.

But the real bug is this sequence:
1. Free `DATEG(last_errors)`
2. Possibly assign stale pointer `*last_errors` to `DATEG(last_errors)`
3. Then free `*last_errors` — which may now be the same as `DATEG(...)` already freed!

Also note: if `last_errors == NULL`, we return early, but if `*last_errors == NULL`, we still proceed and try to access `(*last_errors)->warning_count`.

However, the main issue is **double-free when `DATEG(last_errors) == *last_errors`**.

### Fix Strategy

We must ensure:
1. We never double-free the same pointer.
2. If we transfer ownership (assign to `DATEG`), we don't destroy it afterward.

Key insight: only destroy `*last_errors` if it's **not** the one we just saved to `DATEG`.

So the fix is:
- Only call `timelib_error_container_dtor(*last_errors)` if we did **not** save it into `DATEG`.

This avoids double-free.
