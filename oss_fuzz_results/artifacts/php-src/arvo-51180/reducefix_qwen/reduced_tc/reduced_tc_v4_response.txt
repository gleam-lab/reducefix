### Bug Analysis

The crash is a **heap-use-after-free** occurring in `update_errors_warnings`, specifically during a `READ 4` after calling `timelib_error_container_dtor`. The stack trace shows:

```
timelib_error_container_dtor
php_date_initialize
zim_DateTime___construct
```

This indicates that we are accessing memory after it has been freed.

Looking at the code:
```c
if ((*last_errors)->warning_count || (*last_errors)->error_count) {
    DATEG(last_errors) = *last_errors;
}

timelib_error_container_dtor(*last_errors);
*last_errors = NULL;
```

The issue is:  
We assign `*last_errors` to `DATEG(last_errors)` (a global), and then immediately **free** `*last_errors` with `timelib_error_container_dtor`. This means `DATEG(last_errors)` now points to **freed memory**, which will later be accessed → **use-after-free**.

Moreover, if `last_errors == NULL`, we return early — but not before potentially using `DATEG(last_errors)` which expands to a macro likely referencing thread-local storage.

But more critically: the call to `timelib_error_container_dtor(*last_errors)` frees the container, yet earlier we may have stored it in `DATEG(last_errors)` without taking ownership or copying.

### Root Cause
- `DATEG(last_errors)` is set to point directly to `*last_errors`
- Then `*last_errors` is **freed**
- Later access to `DATEG(last_errors)` uses **dangling pointer**

Additionally, line 2273 checks `last_errors == NULL`, but `DATEG(last_errors)` was already dereferenced above — if `last_errors` is null, this causes undefined behavior.
