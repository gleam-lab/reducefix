The crash is a **heap-use-after-free** in `update_errors_warnings`, triggered via `php_date_initialize` during `DateTime::__construct`. The stack trace shows that after `timelib_error_container_dtor` is called, memory is accessed again.

### 🔍 Root Cause Analysis:

In the function:
```c
static void update_errors_warnings(timelib_error_container **last_errors)
```

We see:
1. If `DATEG(last_errors)` exists, it's freed with `timelib_error_container_dtor(DATEG(last_errors))`
2. Then, later, we conditionally assign: `DATEG(last_errors) = *last_errors;`
3. **But then**: `timelib_error_container_dtor(*last_errors);` — this frees `*last_errors`
4. However, if `*last_errors` was just assigned to `DATEG(last_errors)`, then `DATEG(last_errors)` now points to **freed memory**

This creates a **dangling pointer** in `DATEG(last_errors)` — a classic **use-after-free**.

Specifically:
- Line 2278: `DATEG(last_errors) = *last_errors;`
- Line 2281: `timelib_error_container_dtor(*last_errors);` → frees what `DATEG(last_errors)` points to

Thus, later when `DATEG(last_errors)` is used (e.g., in `timelib_error_container_dtor` in another call), we get **use-after-free READ 4**.

---

### ✅ Fix Strategy:
Do **not** destroy `*last_errors` **if** it has been stored into `DATEG(last_errors)`.

So: only call `timelib_error_container_dtor(*last_errors)` if we did **not** save it to `DATEG(last_errors)`.

We must avoid double-free and dangling pointers.

### ✅ Explanation of Fix:

- If the error container has warnings/errors, we **transfer ownership** to `DATEG(last_errors)` and **do not destroy** it.
- Otherwise, we **destroy** it immediately.
- In both cases, we set `*last_errors = NULL` to avoid dangling pointers.

This prevents the use-after-free because:
- We never free an object that is still referenced by `DATEG(last_errors)`
- `DATEG(last_errors)` now safely owns the pointer if it was set
- No double-free or dangling pointer remains

- Transferred safely to global storage (`DATEG`), or
- Destroyed immediately if not needed
